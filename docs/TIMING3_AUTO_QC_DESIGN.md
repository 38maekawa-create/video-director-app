## 1. 目的

事務さんが素材をアップし、編集者が編集した動画に対して:
1. **テロップの誤字を自動検出**する（編集者の誤字チェック工数を削減）
2. **ハイライト選定・強調演出ポイントの妥当性を自動精査**する（マーケ品質のQC）

## 2. 全体フロー

```
編集者が編集完了 → Vimeoに編集済み動画アップ
    ↓
【トリガー】Vimeo APIで新バージョン検知 or アプリから手動起動
    ↓
【Step 1】Whisper APIで音声文字起こし（= 発言の正解データ）
    ↓
【Step 2】動画フレームを一定間隔でキャプチャ
    ↓
【Step 3】GPT-4oビジョンでテロップ検出＋読み取り
    - テロップが映っているフレームを検出
    - テロップのテキストを読み取り
    - タイムスタンプと紐付け
    ↓
【Step 4】QCチェック2種を実行
    ├── A: テロップ正確性チェック
    │     Whisper文字起こし（正解）vs GPT-4oビジョン（テロップ実物）を突合
    │     不一致箇所をフラグ（誤字・脱字・発言との乖離）
    │
    └── B: ハイライト選定 + 強調演出ポイント精査（Opus定額内）
          QUALITY_JUDGMENT_GUIDE注入:
          - セクション1: ゲスト層分類の判断基準
          - セクション2: 演出ディレクション品質基準
          - セクション3: タイトル設計思想
          - セクション5: ハイライト選定基準（パンチラインの4つの引き金）
          ディレクションレポートの指示内容と突合:
          - 「このハイライトは本当に最強のパンチラインか？」
          - 「この強調演出ポイントは層に合ってるか？」
          - 「ディレクションで指示したポイントが反映されてるか？」
    ↓
【Step 5】アプリ内にQC結果表示
    - プロジェクトごとにQCステータス（✅問題なし / 🔴問題あり）
    - 問題ありの場合: 具体的な指摘事項を一覧表示
      - テロップ誤字: 「[03:24] 発言: キャッシュフロー → テロップ: キャシュフロー」
      - ハイライト指摘: 「冒頭ハイライトのパンチラインが弱い。代替候補: [候補]」
    - 将来拡張: フラグから修正指示まで自動生成（おいおい）
```

## 3. 技術スタック

| コンポーネント | 技術 | コスト |
|---|---|---|
| 音声文字起こし | Whisper API（large-v3） | 従量課金（1動画数十円〜100円程度。許容範囲） |
| テロップ読み取り | GPT-4o Vision API | 従量課金（1動画あたり画像数十枚で数百円程度。許容範囲） |
| マーケQC判定 | Opus（claude -p経由、定額内） | MAX定額内 |
| フレームキャプチャ | ffmpeg（ローカル処理） | 無料 |
| 動画取得 | Vimeo API（既存機能流用） | プラン内 |

## 4. 設計判断の根拠

### なぜOCRではなくGPT-4oビジョンか
- AI開発3（スレッズシステム）でOCR（Tesseract等）を使った際、誤字・文字崩れが多発した実績あり
- GPT-4oビジョンはフォント崩れ・背景ノイズ・装飾文字に強く、テロップ読み取り精度が高い
- コスト差は1動画あたり数百円程度で、精度向上に対して十分見合う

### なぜWhisper APIか（定額内制約の例外）
- 日本語音声認識で最高精度
- Vimeo自動キャプションより精度が高い
- テロップ誤字検出の「正解データ」として使うため、ここの精度が全体のQC品質を決める
- なおとさん承認済み: 「WhisperやGPT-4oくらいならAPI大丈夫」（2026-03-21）

### なぜVimeo文字起こしを使わないか
- Vimeo自動キャプションは英語に強いが日本語精度は低い
- 「正解データ」として使うには精度不足
- ただし、Vimeo文字起こしが既に走っている既存機能との統合は検討余地あり

## 5. フレームキャプチャの設計

### キャプチャ間隔
- 基本: **2秒間隔**でフレームキャプチャ
- テロップは通常2-5秒表示されるため、2秒間隔なら見逃さない
- 30分動画 = 約900フレーム。GPT-4oビジョンに全部投げるとコスト高
- → **前処理でテロップ有無を判定**（画像下部1/3の領域に文字があるかを軽量モデルで判定）して、テロップありフレームだけGPT-4oに投げる
- テロップありフレームは1動画あたり30-80枚程度と推定

### ffmpegコマンド例
```bash
ffmpeg -i input.mp4 -vf "fps=0.5" -q:v 2 frames/frame_%04d.jpg
```

## 6. 既存機能との統合

### 流用可能な既存コンポーネント
- Vimeo API連携（api_server.py）: 動画取得・バージョン管理は既存
- ディレクションレポート（direction_generator.py）: QC突合の基準データ
- QUALITY_JUDGMENT_GUIDE.md: マーケQCのプロンプト注入元
- 承認フロー（approval API）: QC結果の承認/却下UIは流用可能

### 新規実装が必要なコンポーネント
1. `auto_qc_runner.py` — QCパイプラインのメインスクリプト
2. `whisper_transcriber.py` — Whisper API呼び出し+タイムスタンプ付き文字起こし
3. `frame_extractor.py` — ffmpegフレームキャプチャ+テロップ有無前処理
4. `telop_reader.py` — GPT-4oビジョンによるテロップ読み取り
5. `qc_comparator.py` — 正解データ vs テロップの突合+フラグ生成
6. `marketing_qc.py` — QUALITY_JUDGMENT_GUIDE注入+マーケ的QC判定
7. iOS側: QC結果表示画面（プロジェクト詳細にQCタブ追加）
8. API: QC結果取得エンドポイント
